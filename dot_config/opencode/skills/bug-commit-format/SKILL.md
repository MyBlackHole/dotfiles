# Bug 修复提交格式

## 概述

Bug 修复的提交信息必须包含完整的问题分析，确保后续可追溯和审查。

**核心原则：** 每次 bug 修复提交必须包含问题描述、原因分析、解决方案、影响范围、性能影响、复现步骤、测试验证、相关 Issue、风险评估、回滚方案十个核心要素。

## 适用场景

**始终使用：**
- 修复 bug 后的代码提交
- 热修复或补丁提交
- 缺陷修复相关的任何提交

**此规则适用于所有 bug 修复，无论：**
- Bug 严重程度
- 影响范围大小
- 修复复杂度

## 铁律

```
无完整分析则无 Bug 修复提交
无版本号更新则无 Bug 修复提交
```

提交信息缺少任何核心要素？停止。补充完整后再提交。

**省略意味着什么：**
- 不跟版本号 = 不知道哪个版本修复的 = 无法追溯
- 不做风险评估 = 可能引入更大问题
- 不写回滚方案 = 出问题无法快速恢复

## 提交信息模板

### 标题格式

```
【B-XXXX】模块: 简短描述, OLD_VERSION -> NEW_VERSION
```

- `【B-XXXX】`：Bug 单号，使用中文全角括号
- `模块`：修改的模块/组件名（如 rpc、fsdeamon、libobk）
- `简短描述`：一句话概括修复内容
- `OLD_VERSION -> NEW_VERSION`：本次涉及的版本号变更，写在标题上便于 git log --oneline 直接看到

### 标准格式

```
【B-XXXX】模块: 简短描述, OLD_VERSION -> NEW_VERSION

问题描述: <描述 bug 现象和复现步骤>
原因分析: <分析根本原因>
解决方案: <描述修复方案>
影响范围: <描述影响的功能/模块>
性能影响: <分析性能影响（有/无/需测试）>
复现步骤: <详细复现步骤>
测试验证: <测试方式和结果>
相关 Issue: <关联的 Issue ID>
风险评估: <引入的新风险>
回滚方案: <如何撤销此次修复>
版本变更: <列出所有版本号变更，如有>
```

### 简化格式（适用于简单修复）

```
Bug 修复: <简短描述>

问题: <1-2句话>
原因: <根本原因>
方案: <修复方法>
影响: <影响范围>
```

### 示例

**完整格式：**

```
【B-123】rpc: 修复RPC重连后remote_dir_fd失导致EBADF, 3.6.3.10 -> 3.6.3.11

问题描述: backup full模式执行中，单次RPC连接超时重连后，后续操作失败报EBADF(errno:9)，任务级联失败。
原因分析: backup_thread_ctx缓存了服务端侧fd编号。RPC重连时旧连接被释放(close dir_fd)，新连接dir_fd=-1，但客户端仍复用旧fd编号。
解决方案: 将openat+download打包为RPC函数，由rpc_conn自管理remote fd缓存——重连时清缓存，下次调用自动reopen。
影响范围: backup-client.cpp的backup_fileat_thread/linkat_thread，不影响路径版API和restore。
性能影响: 对NFS保留dir fd缓存，重连仅多一次openat RPC(几毫秒)。
复现步骤: 1. 大文件backup 2. RPC超时 3. 重连后所有文件报EBADF
测试验证: 新增download_file_dir测试覆盖缓存命中/未命中/清缓存重开三场景，通过。
相关 Issue: 1951
风险评估: 低。重连清缓存与原有rpc_conn_restart机制一致。
回滚方案: revert本次commit，恢复原openat逻辑。
版本变更: xmake.lua: 3.6.3.10 -> 3.6.3.11
```

**简化格式：**

```
Bug修复: 修复数组越界崩溃

问题: 极端情况下索引超出数组边界
原因: 未对边界条件做校验
方案: 添加边界检查和默认值处理
影响: 仅影响导出功能，错误概率<0.1%
```

## 核心要素说明

### 1. 标题（最关键）

必须使用项目规范格式：
- Bug 修复：`【B-XXXX】模块: 描述, VER_OLD -> VER_NEW`
- 功能开发：`【F-XXXX】模块: 描述, VER_OLD -> VER_NEW`

### 2. 问题描述

必须包含：
- Bug 现象
- 复现步骤（关键路径）
- 影响的用户场景

### 2. 原因分析

必须包含：
- 根本原因（不是症状）
- 触发条件
- 相关代码位置

### 3. 解决方案

必须包含：
- 修复方法
- 修改的文件和函数
- 替代方案（如果有）

### 4. 影响范围

必须包含：
- 功能模块
- 受影响的用户/场景
- 风险等级

### 5. 性能影响

必须包含：
- 有/无性能影响
- 预期影响值（如有）
- 是否需要压测

### 6. 复现步骤

必须包含：
- 详细的复现步骤（1、2、3...）
- 必要的测试数据
- 环境条件

### 7. 测试验证

必须包含：
- 测试方式（单元测试/集成测试/手动测试）
- 测试结果
- 验证结论

### 8. 相关 Issue

必须包含：
- 关联的 Issue/Bug 单号
- 相关 MR/PR（如果有）

### 9. 风险评估

必须包含：
- 修复可能引入的新风险
- 风险等级
- 缓解措施

### 11. 版本变更

必须包含：
- 所有变更的版本号（旧 -> 新）
- 版本号所在文件（如 xmake.lua、version.h 等）
- 版本号已在标题中体现，正文中列出详情

## 常见借口

| 借口 | 事实 |
|--------|---------|
| "只改了几行代码不用升版本" | 任何行为变更都需要版本号追溯 |
| "太简单不需要" | 简单也需要记录，便于追溯 |
| "紧急，没时间" | 紧急不是省略的理由 |
| "其他人看得懂" | 一个月后你自己也看不懂 |
| "已经有issue了" | Issue 和提交信息不重复 |
| "代码就是最好的文档" | 代码不说明为什么这么改 |

## 红色警示 - 停止

- 提交信息只有 "fix"、"bug fix"
- 标题格式不是 `【B-XXXX】` 开头
- 标题缺少版本号变更 `OLD -> NEW`
- 缺少原因分析
- 缺少影响范围
- 使用 "WIP"、"TODO" 等标记
- 复制错误日志作为原因

**以上均意味着：补充完整信息后再提交。**

## 验证检查清单

完成 bug 修复提交前：

- [ ] 标题格式 `【B-XXXX】模块: 描述, VER_OLD -> VER_NEW`
- [ ] 已更新版本号并提交了版本文件
- [ ] 包含问题描述
- [ ] 包含原因分析（根本原因，非症状）
- [ ] 包含解决方案
- [ ] 包含影响范围
- [ ] 包含性能影响评估
- [ ] 包含复现步骤
- [ ] 包含测试验证
- [ ] 包含相关 Issue
- [ ] 包含风险评估
- [ ] 包含回滚方案
- [ ] 包含版本变更（在标题和正文中）

缺少任何项？补充完整。

## 自动化辅助

### Git hooks 配置

可添加 pre-commit hook 验证提交信息格式：

```bash
#!/bin/bash
# 检查提交信息是否包含必需字段
COMMIT_MSG=$(cat "$1")
if ! echo "$COMMIT_MSG" | grep -q "问题描述:"; then
    echo "错误: 提交信息缺少'问题描述'字段"
    exit 1
fi
# 可扩展更多检查...
```

### 提交模板

在项目根目录添加 `.git-commit-template`：

```
【B-XXXX】模块: 简短描述, VER_OLD -> VER_NEW

问题描述: 
原因分析: 
解决方案: 
影响范围: 
性能影响: 
复现步骤: 
测试验证: 
相关 Issue: 
风险评估: 
回滚方案: 
版本变更: 
```

## 总结

| 要素 | 必需 | 说明 |
|------|------|------|
| 标题格式 | ✅ | `【B-XXXX】模块: 描述, VER_OLD -> VER_NEW` |
| 版本变更 | ✅ | 更新版本号并在标题中体现 |
| 问题描述 | ✅ | Bug 现象和复现步骤 |
| 原因分析 | ✅ | 根本原因，非症状 |
| 解决方案 | ✅ | 修复方法和修改位置 |
| 影响范围 | ✅ | 功能模块和风险等级 |
| 性能影响 | ✅ | 有/无及预期值 |
| 复现步骤 | ✅ | 详细复现路径 |
| 测试验证 | ✅ | 测试方式和结果 |
| 相关 Issue | ✅ | 关联的 Issue ID |
| 风险评估 | ✅ | 新风险和缓解措施 |
| 回滚方案 | ✅ | 撤销步骤和位置 |

无论大小 bug，均需完整记录。版本号是追溯修复的唯一标识。
