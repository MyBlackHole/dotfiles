# 需求开发提交格式

## 概述

需求开发的提交信息必须包含完整的实现分析，确保需求可追溯和审查。

**核心原则：** 每次需求开发提交必须包含需求描述、需求背景、实现方案、影响范围、测试验证、相关需求、性能影响、风险评估、回滚方案、验收标准十个核心要素。

## 适用场景

**始终使用：**
- 新功能开发后的代码提交
- 需求实现相关的任何提交
- 功能迭代或扩展开发

**此规则适用于所有需求开发，无论：**
- 需求大小
- 复杂程度
- 紧急程度

## 铁律

```
无完整分析则无需求开发提交
无版本号更新则无需求开发提交
```

提交信息缺少任何核心要素？停止。补充完整后再提交。

**无例外：**
- 不要因为"简单功能"而省略
- 不要因为"紧急"而跳过
- 不要因为"小需求"而简化

完整分析是代码质量的一部分。

## 提交信息模板

### 标题格式

```
【F-XXXX】模块: 简短描述, OLD_VERSION -> NEW_VERSION
```

- `【F-XXXX】`：需求单号，使用中文全角括号
- `模块`：修改的模块/组件名
- `简短描述`：一句话概括实现的功能
- `OLD_VERSION -> NEW_VERSION`：版本号变更，写在标题上便于 git log --oneline 直接看到

```
[需求ID] 简短标题

需求描述: <描述实现的功能>
需求背景: <描述为什么需要此功能>
实现方案: <描述具体实现方法和设计>
影响范围: <描述影响的功能/模块/用户>
测试验证: <描述测试方式和结果>
相关需求: <关联的需求ID（前置/后置/依赖）>
性能影响: <分析性能影响（有/无/需测试）>
风险评估: <可能的风险和缓解措施>
回滚方案: <如何撤销此次变更>
验收标准: <如何验证功能是否满足要求>
```

### 简化格式（适用于简单需求）

```
需求: <简短标题>

描述: <1-2句话描述>
背景: <为什么需要>
方案: <实现方法>
影响: <影响范围>
测试: <测试结果>
```

### 示例

**完整格式：**

```
[FEAT-456] 新增用户权限管理模块

需求描述: 实现基于角色的权限管理功能，支持角色创建、权限分配、权限继承
需求背景: 现有系统权限管理过于简单，无法满足精细化权限控制需求，多个业务线反馈权限配置不灵活
实现方案: 1. 新增角色管理表和权限关联表 2. 实现角色CRUD接口 3. 添加权限校验中间件 4. 提供管理后台界面
影响范围: 用户管理模块、权限校验模块（/auth/*、/user/*）
测试验证: 单元测试通过（35个），集成测试通过，手动测试通过，覆盖主要场景
相关需求: REQ-300（用户模块基础功能）、REQ-450（审计日志）
性能影响: 新增3张表，查询性能影响<5%，需压测验证
风险评估: 权限变更可能影响现有用户，需提前通知并提供旧版兼容
回滚方案: 回滚至上一个版本，新增的表和接口在 v1.2.0 标签
验收标准: 1. 支持角色创建/编辑/删除 2. 支持权限分配 3. 支持权限继承 4. 管理后台可配置
```

**简化格式：**

```
需求: 新增导出CSV功能

描述: 支持将列表数据导出为CSV文件
背景: 用户反馈需要批量导出数据进行分析
方案: 新增导出接口，使用csv库生成文件
影响: 仅影响数据列表页，使用率<10%
测试: 单元测试通过，手动测试1000行数据导出正常
```

## 核心要素说明

### 1. 需求描述

必须包含：
- 实现的功能名称
- 核心功能点
- 用户价值

### 2. 需求背景

必须包含：
- 需求来源（用户反馈、业务需求、技术债）
- 解决的问题
- 业务价值

### 3. 实现方案

必须包含：
- 技术方案选择
- 数据模型变更
- 接口设计
- 关键实现点

### 4. 影响范围

必须包含：
- 功能模块
- 受影响的用户/场景
- 依赖方通知

### 5. 测试验证

必须包含：
- 测试方式（单元/集成/手动）
- 测试覆盖率
- 测试结果

### 6. 相关需求

必须包含：
- 关联的需求ID
- 前置需求
- 依赖需求

### 7. 性能影响

必须包含：
- 有/无性能影响
- 预期影响值（如有）
- 是否需要压测

### 8. 风险评估

必须包含：
- 可能的风险
- 风险等级
- 缓解措施

### 9. 回滚方案

必须包含：
- 回滚的具体步骤
- 涉及的文件和数据
- 回滚后的验证方式

### 10. 验收标准

必须包含：
- 功能验收点
- 非功能验收点
- 验证方法

## 常见借口

| 借口 | 事实 |
|--------|---------|
| "太简单不需要" | 简单也需要记录，便于追溯 |
| "紧急，没时间" | 紧急不是省略的理由，可以简化格式 |
| "其他人看得懂" | 一个月后你自己也看不懂 |
| "已经有需求文档了" | 需求文档和提交信息不重复 |
| "代码就是最好的文档" | 代码不说明为什么这么实现 |

## 红色警示 - 停止

- 提交信息只有 "feat"、"new feature"
- 缺少需求背景
- 缺少实现方案
- 缺少影响范围
- 使用 "WIP"、"TODO" 等标记
- 复制需求文档作为描述

**以上均意味着：补充完整信息后再提交。**

## 验证检查清单

完成需求开发提交前：

- [ ] 包含需求描述
- [ ] 包含需求背景
- [ ] 包含实现方案
- [ ] 包含影响范围
- [ ] 包含测试验证
- [ ] 包含相关需求
- [ ] 包含性能影响评估
- [ ] 包含风险评估
- [ ] 包含回滚方案
- [ ] 包含验收标准
- [ ] 使用规范的标题格式

缺少任何项？补充完整。

## 自动化辅助

### Git hooks 配置

可添加 pre-commit hook 验证提交信息格式：

```bash
#!/bin/bash
# 检查提交信息是否包含必需字段
COMMIT_MSG=$(cat "$1")
if ! echo "$COMMIT_MSG" | grep -q "需求描述:"; then
    echo "错误: 提交信息缺少'需求描述'字段"
    exit 1
fi
# 可扩展更多检查...
```

### 提交模板

在项目根目录添加 `.git-commit-template`：

```
[需求ID] 简短标题

需求描述: 
需求背景: 
实现方案: 
影响范围: 
测试验证: 
相关需求: 
性能影响: 
风险评估: 
回滚方案: 
验收标准: 
```

## 总结

| 要素 | 必需 | 说明 |
|------|------|------|
| 需求描述 | ✅ | 实现的功能和核心功能点 |
| 需求背景 | ✅ | 需求来源和业务价值 |
| 实现方案 | ✅ | 技术方案和关键实现 |
| 影响范围 | ✅ | 功能模块和用户影响 |
| 测试验证 | ✅ | 测试方式和结果 |
| 相关需求 | ✅ | 关联的需求ID |
| 性能影响 | ✅ | 有/无及预期值 |
| 风险评估 | ✅ | 风险和缓解措施 |
| 回滚方案 | ✅ | 撤销步骤和位置 |
| 验收标准 | ✅ | 功能和非功能验收点 |

无论大小需求，均需完整记录。
